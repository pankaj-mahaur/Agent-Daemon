# engineering/

Daily code work. These skills fire when you're debugging, implementing, reviewing, or applying a methodology to a real change.

| Skill | What it does |
|---|---|
| [audit-runner](audit-runner/) | Execute a list of findings against a codebase chunk-by-chunk with severity sequencing + single-concern commits |
| [db-migrations](db-migrations/) | Database schema and migration discipline (safety, backfill, rollback) |
| [dead-code-review](dead-code-review/) | Proof-based dead code cleanup |
| [debug-triage](debug-triage/) | Triage ladder (services → data → cache → request → code) + bug-class catalog + disciplined diagnosis loop |
| [deploy-ops](deploy-ops/) | Deploy contract, CI gates, env-var plumbing, rollback paths |
| [diagnose-fetch-failure](diagnose-fetch-failure/) | "CORS error / ERR_FAILED" — almost always a backend 5xx with stripped CORS headers |
| [diagnose-intermittent-failure](diagnose-intermittent-failure/) | "Sometimes works, sometimes doesn't" — local backend flakes |
| [docs-sync-audit](docs-sync-audit/) | Detect and fix documentation drift |
| [implement-feature](implement-feature/) | Search-before-you-write, match-the-neighbor, one-commit-per-root-cause |
| [llm-app-safety](llm-app-safety/) | LLM application safety (prompt injection, output validation, jailbreaks) |
| [merge-feature-branch](merge-feature-branch/) | Safely merge a shared branch into a long-lived feature branch |
| [methodology-api-design](methodology-api-design/) | API design discipline — versioning, evolvability, error shapes |
| [methodology-architectural-decision](methodology-architectural-decision/) | Capture architectural decisions as ADRs |
| [methodology-brainstorm](methodology-brainstorm/) | Generate-then-prune approach to ideation |
| [methodology-code-review](methodology-code-review/) | Code review discipline — what to look for, what to skip |
| [methodology-dependency-management](methodology-dependency-management/) | Manage third-party dependencies with intent |
| [methodology-documentation](methodology-documentation/) | Write docs that survive — sources, scope, freshness |
| [methodology-error-handling](methodology-error-handling/) | Error handling that surfaces failures, not swallows them |
| [methodology-incremental-delivery](methodology-incremental-delivery/) | Ship in slices, not waterfalls |
| [methodology-pair-programming](methodology-pair-programming/) | Pair programming with an LLM — driver/navigator, when to switch |
| [methodology-performance-profiling](methodology-performance-profiling/) | Measure first, optimize second |
| [methodology-refactoring](methodology-refactoring/) | Refactor under green tests — extract, deepen, simplify |
| [methodology-systematic-debugging](methodology-systematic-debugging/) | A debugging system, not gut feeling |
| [methodology-tdd](methodology-tdd/) | Red-green-refactor + deep modules + DI + SDK-style mocking |
| [methodology-writing-plan](methodology-writing-plan/) | Plan-before-code — produce prep docs before any non-trivial feature |
| [multiplatform-parity](multiplatform-parity/) | Keep web/iOS/Android in functional parity through change |
| [optimization-audit](optimization-audit/) | Frontend + backend performance review |
| [production-readiness](production-readiness/) | Launch readiness checklist across all layers |
| [review-slice](review-slice/) | Deep-review a page/feature using a 9-class bug checklist |
| [security-audit](security-audit/) | Security review with trust boundary mapping |
| [seed-data](seed-data/) | Generate idempotent database seed scripts with realistic test data |
